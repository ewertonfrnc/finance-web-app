finance-api — Discovery para integração com o finance-web-app

Backend Go/Postgres · gerado 2026-06-20 · foco: o que o backend já computa e o que o front pode deixar de calcular
Conclusão em uma linha: o finance-api já calcula saldo diário acumulado, carry-forward entre meses (com cache), projeção de dias futuros, expansão de recorrência (daily/weekly/monthly/yearly + exceções + edição por escopo), o valor do "diário" e tags. O web-app hoje refaz quase tudo isso no cliente sobre finance.json + localStorage. Integrar = apagar essa camada local e consumir a API.

4. Fluxo do /v1/balance (o que substitui o cálculo local)

GET /v1/balance?year=2026&month=6
  │ requireAuth → userID
  │ DailyBudgetConfigs.GetConfigAt(userID, 1º do mês) → config.Amount (o "diário", versionado por valid_from)
  │ BalanceCache.Get(mês anterior) → carryForward (cache hit)
  │   else Transactions.GetNetBalanceBefore(1º do mês) → soma tudo antes (expande recorrência em SQL)
  │ adjustCarryForward → desconta diário projetado se o mês é futuro
  │ Transactions.GetDailyTotals(from,to) → totais por dia/tipo (recorrência expandida via generate_series + exceptions)
  │ computeMonthlyBalance → para cada dia: running += income − expense − (dailyActual + dailyProjected) − savings
  │   dailyProjected = config.Amount se dia > hoje (senão 0)
  │ se mês já fechou → BalanceCache.Set
  ▼ []DayBalance{ day, income, expense, daily, daily_projected, savings, running_balance }
Runway dos mockups = backend, não constante. O ≈ N dias = running_balance ÷ daily_budget.amount. O R$ 53,33 do prompt é só o exemplo; o valor real é config.Amount de /v1/budget/preview. O front só precisa dividir e colorir.

7. Lacunas e decisões para a integração

⚠ Balance é só mensal

Trilho/Foco mostram 12 meses. Opções: (a) 12× GET /balance em paralelo (TanStack Query, cache por mês) — funciona já; (b) adicionar GET /balance?year= retornando os 12 meses (previsto no doc 06). Recomendo (a) agora, (b) como follow-up de backend.

✗ Web-app não tem auth

API exige JWT. O app é local-only hoje (sem login/registro). Integrar exige client HTTP com token, telas de login/registro/onboarding e storage do token. Mudança grande — ver decisão abaixo.

? Tags: entidade vs string

O plano do web-app tratava etiqueta como string. O backend tem tags como entidade (id, cor, M:N). Os chips "Assinatura/Fixo/Bônus/Reembolso" viram tags reais (criar/selecionar via /tags).

⚠ Contratos a alinhar

  • Tipos plural↔singular: entradas↔entrada etc.
  • Centavos no fio (já há toCents/fromCents).
  • Desempacotar .data do envelope.
  • Datas YYYY-MM-DD (cuidado com DATE→RFC3339 do lib/pq, já tratado no server).
  • CORS aceita http://localhost:*.
Decisão de arquitetura (precisa do seu ok): integrar a API é trocar a fundação do web-app de local-only para cliente autenticado. Caminhos: (1) Cutover total — remove JSON/localStorage, adiciona auth, tudo via API (mais limpo, mais trabalho, exige backend rodando); (2) Incremental — camada de serviço com flag, troca hook a hook (useFinanceYear → /balance primeiro), mantém fallback local; (3) Adiar — segue o plano local-only e integra depois. Isto está registrado no plano (folga-desktop-plan.md, nova Fase 0).

8. Gotchas observados no código

  • lib/pq DATE → RFC3339: coluna DATE escaneada como string volta "...T00:00:00Z"; parseBalanceDate tem fallback (regressão já conhecida do projeto).
  • Cache de saldo (month_balance_cache) é invalidado a partir do mês da transação a cada create/update/delete — o front não precisa se preocupar com staleness.
  • Edição recorrente usa transaction_exceptions + source_series_id/source_occurrence_date: modelo bem mais robusto que o seriesId simples do localStorage atual.
  • adjustCarryForward: ao abrir um mês futuro, o saldo inicial já vem descontado da projeção diária acumulada — replicar isso no front seria fácil de errar; mais um motivo para confiar no server.
  • docs/06 descreve rotas /accounts/:accountId que não existem; ignore, use api.go.
