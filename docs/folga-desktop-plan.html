1.A · O backend já existe: integração com finance-api

Conclusão: o finance-api (Go/Postgres) já computa no servidor quase tudo que o web-app recalcula no cliente. O web-app hoje é local-only (finance.json + localStorage); integrar = apagar essa camada e consumir a API. Discovery: finance-api-discovery.html.

O backend já entrega — não recalcular

  • GET /v1/balance?year=&month= → []DayBalance com saldo acumulado, carry-forward (cache) e projeção futura (daily_projected).
  • Recorrência daily/weekly/monthly/yearly + exceções + edição/exclusão por escopo single|following|all.
  • O "diário" versionado: /v1/budget/preview → daily_budget.amount. Runway = running_balance ÷ amount (o R$ 53,33 é exemplo).
  • Tags como entidade (id+cor, M:N): /v1/tags, PUT /v1/transactions/{id}/tags.
  • Auth JWT; dinheiro int64 centavos; tipos PT-BR singular; envelope {success,data}.

Remover do front — cálculo duplicado

  • lib/finance.ts: calcularSaldosMes, calcularSaldoInicialMes, calcularTotaisMes.
  • service/financeService.ts, localStorageAdapter.ts, transactionStorage.ts, data/normalizeFinance.ts, data/finance.json.
  • useTransactions.buildOccurrences e removeTransactionsFromSeries.
  • useOnboarding (diário ÷30) → register/budget.

Fica no front: saldoColour, sparkline, pico/vale, formatBRL, weekday, e toda a UI.

Lacunas & contratos a alinhar

  • Balance é mensal → 12 meses = 12× GET /balance em paralelo (cache por mês). Recomendado.
  • Web-app não tem auth → client HTTP + token + telas de login/registro/onboarding. Maior peça nova.
  • Tags entidade vs string: chips viram tags reais via /v1/tags.
  • Alinhar plural↔singular, centavos (há toCents/fromCents), desempacotar .data, datas YYYY-MM-DD.
★ Decisão de arquitetura (precisa do seu ok): integrar troca a fundação de local-only para cliente autenticado. (1) Cutover total · (2) Incremental (camada de serviço com flag, troca hook a hook, fallback local — recomendado, começando por useFinanceYear → /v1/balance) · (3) Adiar (segue local-only). Detalhado na Fase 0.

2 · Mudanças de fundação

§2.1 (runway) e §2.3 (preferências) valem em qualquer cenário. §2.2 (tipos/seed/recorrência local) só se ficar local-only; sob integração é substituído pelo backend.

▢ P0  2.1 · lib/finance.ts — modo runway

  • runwayDias(saldo, diario) = saldo / diario — diario vem do backend (daily_budget.amount); local-only usa o do onboarding.
  • saldoColour(saldo, {mode, dark, future}) → {fill, ink}: mode:'runway'|'tier', runway é padrão. Gradiente RGB de 5 stops (≤−10 → 0 → 10 → 22 → ≥40 dias), espelhado p/ dark; future dessatura fill ~42% e ink ~30%.
  • Manter getSaldoColor como adaptador do modo tier; migrar DayRow para saldoColour.
  • Helpers p/ Toolbar/Foco: saldoFimMes, picoVale, serieSaldoAno.

▢  2.2 · Tipos e dados (só local-only)

  • types/transaction.ts: tag?: string; recorrência ganha yearly.
  • buildOccurrences trata yearly + "Termina em" (opcional).
  • Seed de demo no finance.json (Salário/Aluguel/...); remover cartao órfão.

▢  2.3 · Preferências

  • Persistir finance_pref_layout, finance_pref_density, finance_pref_saldo_mode.
  • Default layout='trilho', saldo_mode='runway'.
  • Trilho compacto · Foco confortável.
